# แผนพัฒนาปรับปรุง PageSpeed (รออนุมัติ)

**เป้าหมาย:** ปรับปรุงประสิทธิภาพมือถือ, Best Practices, SEO ตามรายงาน PageSpeed โดยไม่ทำให้ Google AdSense, Clerk หรือฟีเจอร์เดิมพัง  
**สถานะ:** รออนุมัติ — หลังอนุมัติจะทำตามลำดับขั้นตอนด้านล่าง

---

## หลักการ

- **แก้เฉพาะส่วนที่ระบุในแผน** ไม่แก้ logic หลัก, flow ชำระเงิน, webhook, auth
- **ไม่ลบ/จำกัด domain AdSense ใน CSP** และไม่เปลี่ยนวิธีโหลดสคริปต์โฆษณา
- **ทำทีละกลุ่ม → build ผ่าน → ทดสอบ (และถ้าต้อง revert ได้)**  
- **CSP/COOP:** ถ้าทดสอบแล้วกระทบ login หรือโฆษณา จะไม่บังคับหรือ revert

---

## ขั้นที่ 1 — แก้ลิงก์ไม่มีข้อความอธิบาย (SEO) + แคช Static (ความเสี่ยงต่ำ)

### 1.1 ลิงก์ไม่มีข้อความอธิบาย (1 จุด)

| รายการ | การดำเนินการ |
|--------|----------------|
| **วัตถุประสงค์** | แก้ปัญหา "ลิงก์ไม่มีข้อความอธิบาย — พบ 1 ลิงก์" จาก Lighthouse SEO |
| **วิธี** | รัน Lighthouse (หรือเปิด DevTools → Accessibility) บนหน้าแรกและหน้าหลักเพื่อระบุลิงก์ที่ไม่มีข้อความบรรยาย (เช่น ลิงก์ที่มีแต่ไอคอน, ข้อความซ้ำกันมาก, หรือ aria-label ว่าง). จากนั้นแก้เฉพาะจุดนั้น: เพิ่มข้อความที่มองเห็นได้ หรือ `aria-label` ที่มีความหมาย (และ `alt` ถ้าเป็นรูป) |
| **ไฟล์ที่อาจแตะ** | ตามที่ตรวจเจอ (เช่น `components/GeneratorForm.tsx`, `components/CreditsNav.tsx`, `components/Navbar.tsx`, `components/Footer.tsx` หรือ component อื่นที่มี `<a>`/`<Link>`) — **เฉพาะ 1 จุด** |
| **การตรวจสอบ** | รัน Lighthouse SEO หลังแก้ — ต้องไม่มี "ลิงก์ไม่มีข้อความอธิบาย" |

### 1.2 แคชของ Static Assets

| รายการ | การดำเนินการ |
|--------|----------------|
| **วัตถุประสงค์** | "ใช้อายุการใช้งานแคชที่มีประสิทธิภาพ" — ประหยัด ~18 KiB, เร็วขึ้นเมื่อผู้ใช้กลับมาเข้าชม |
| **วิธี** | ใน `next.config.js` ใช้ `headers()` เพิ่ม header `Cache-Control` สำหรับ path `/_next/static/*` (และถ้ามี `/_next/image` ที่เป็น static) ให้มี `public, max-age=31536000, immutable` เพราะ Next สร้างชื่อไฟล์แบบมี hash อยู่แล้ว. **ไม่ตั้งแคช** ให้ path เช่น `/api/*`, `/*.json` หรือหน้าที่ต้อง fresh |
| **ไฟล์ที่แตะ** | `next.config.js` เท่านั้น (เพิ่ม rule ใน `headers()`) |
| **การตรวจสอบ** | Build ผ่าน, เปิดเว็บแล้วตรวจใน DevTools → Network ว่า response ของ `_next/static/*` มี `Cache-Control: public, max-age=31536000, immutable` (หรือเทียบเท่า) |

## ขั้นที่ 2 — ลด Unused JS และ Render-blocking (Performance)

### 2.1 โหลด component ที่ไม่จำเป็นในโหลดแรกแบบ lazy

| รายการ | การดำเนินการ |
|--------|----------------|
| **วัตถุประสงค์** | ลด Unused JavaScript (~448 KiB) และงานบน main thread — โหลดเฉพาะสิ่งที่ต้องใช้สำหรับ above-the-fold ก่อน |
| **วิธี** | ตรวจสอบ `app/layout.tsx` และหน้าแรก `app/page.tsx`: component ใดที่ไม่ได้แสดงทันที (เช่น modal, consent, บล็อกด้านล่าง) พิจารณาโหลดแบบ dynamic ด้วย `next/dynamic` และ `ssr: false` เฉพาะที่เหมาะสม (เช่น `CreditModal`, `CookieConsent` โหลดหลัง hydration ได้). **ไม่** lazy โหลด `ClerkProvider`, `Navbar`, `Footer`, หรือสคริปต์ AdSense ใน `<head>` |
| **ไฟล์ที่อาจแตะ** | `app/layout.tsx` (dynamic import สำหรับ `CreditModal`, `CookieConsent` ถ้าไม่กระทบการแสดงผล), หน้าแรกถ้ามี component ใหญ่ที่อยู่ล่าง fold |
| **การตรวจสอบ** | Build ผ่าน, หน้าแรกและหน้าหลักเปิดได้, Sign-in / Sign-up ทำงาน, Cookie consent แสดงและ Accept/Reject ได้, โฆษณาแสดง (ถ้ามี slot) |

### 2.2 สคริปต์ AdSense (คงไว้เหมือนเดิม)

| รายการ | การดำเนินการ |
|--------|----------------|
| **วัตถุประสงค์** | ไม่ให้โฆษณาพัง |
| **วิธี** | **ไม่เปลี่ยน** การโหลดสคริปต์ AdSense ใน `app/layout.tsx` (คง `async` + `crossOrigin="anonymous"` ตามเดิม). ไม่ย้ายไปโหลดหลัง LCP ในขั้นนี้ เพราะอาจทำให้โฆษณาไม่แสดงหรือผิดนโยบาย |

## ขั้นที่ 3 — ปรับ LCP บนมือถือ (Performance)

### 3.1 เร่งการแสดงองค์ประกอบ LCP

| รายการ | การดำเนินการ |
|--------|----------------|
| **วัตถุประสงค์** | ลด LCP จาก ~4.0s ให้เข้าเกณฑ์ "ดี" (< 2.5s) บนมือถือ |
| **วิธี** | (1) หน้าแรก: LCP มักเป็นข้อความหัวข้อ "Create Viral Hooks..." หรือ hero section — ไม่มีรูป hero ขนาดใหญ่ในโค้ดปัจจุบัน จึงโฟกัสที่การลดการรอจาก JS/CSS. (2) ถ้ามี `<img>` ใดใน above-the-fold ของหน้าแรก ให้เพิ่ม `priority` หรือ `fetchPriority="high"`. (3) ไม่เพิ่มรูปใหม่. (4) พึ่งผลจากขั้นที่ 2 (ลด JS ที่โหลดก่อน LCP) เป็นหลัก |
| **ไฟล์ที่อาจแตะ** | `app/page.tsx` (เฉพาะ attribute ของรูป ถ้ามี), หรือไม่แตะถ้า LCP เป็นข้อความและขั้นที่ 2 ลด blocking ได้เพียงพอ |
| **การตรวจสอบ** | Build ผ่าน, รัน PageSpeed Insights มือถืออีกครั้ง — เป้าหมาย LCP < 2.5s (หรือดีขึ้นจากเดิม) |

## ขั้นที่ 4 — Best Practices (คอนโซล / Issues)

### 4.1 ข้อผิดพลาดคอนโซลและ Issues ใน DevTools

| รายการ | การดำเนินการ |
|--------|----------------|
| **วัตถุประสงค์** | แก้ "บันทึกข้อผิดพลาดเบราว์เซอร์ลงในคอนโซล" และ "บันทึกปัญหาไว้แล้วในแผง Issues" เพื่อความเสถียรและคะแนน Best Practices |
| **วิธี** | เปิดเว็บใน Chrome → F12 → Console และแผง Issues; ระบุ error/warning ที่มาจากโค้ดเรา (ไม่ใช่แค่ third-party). แก้เฉพาะจุดที่แก้ได้โดยไม่เปลี่ยนพฤติกรรม (เช่น ใส่ dependency ที่ขาด, แก้ type เล็กน้อย). ถ้าเป็น warning จาก Clerk/AdSense อย่างเดียว จะไม่แก้โค้ดเรา |
| **ไฟล์ที่อาจแตะ** | ตามที่ตรวจเจอ (อาจเป็น component ที่ใช้ hook หรือ API ผิดพลาด) |
| **การตรวจสอบ** | Build ผ่าน, เปิดหน้าหลักและ tools แล้ว Console/Issues ไม่มี error ใหม่จากโค้ดเรา |

---

## ขั้นที่ 5 — ความปลอดภัย (CSP / COOP) — ทำเมื่อขั้น 1–4 เสร็จ และทดสอบเข้มงวด

### 5.1 CSP และ COOP

| รายการ | การดำเนินการ |
|--------|----------------|
| **วัตถุประสงค์** | "ตรวจสอบว่า CSP มีผลกับการโจมตี XSS" และ "รับรองการแยกต้นทางด้วย COOP" ตาม PageSpeed |
| **วิธี** | (1) **CSP:** ไม่ลบหรือจำกัด domain ที่เกี่ยวกับ AdSense และ Clerk (คง `pagead2.googlesyndication.com`, `tpc.googlesyndication.com`, `*.googlesyndication.com`, `*.g.doubleclick.net`, `googleads.g.doubleclick.net`, และ Clerk ตามเดิม). ถ้าจะลด `unsafe-inline` ต้องทดสอบทุกหน้าที่มีฟอร์มและโฆษณา. (2) **COOP:** เพิ่ม header `Cross-Origin-Opener-Policy: same-origin` (หรือ `same-origin-allow-popups` ถ้า Clerk ใช้ popup) ใน `next.config.js` แล้วทดสอบว่า Sign-in/Sign-up และการ redirect ยังทำงาน. ถ้าไม่ทำงาน จะใช้ค่าเดิมหรือ revert |
| **ไฟล์ที่แตะ** | `next.config.js` (headers) |
| **การตรวจสอบ** | หลัง deploy หรือรัน production build: Sign-in, Sign-up, ลงชื่อออก, หน้า pricing, หน้า tools ที่มีโฆษณา ทำงานปกติ; โฆษณาแสดง (ถ้ามี slot); ไม่มี error ในคอนโซลจาก CSP/COOP |

### 5.2 สิ่งที่ไม่ทำในแผนนี้

- **Trusted Types:** ไม่เพิ่มในแผนนี้ (เสี่ยงกระทบ third-party และต้องทดสอบมาก)
- **การลบ/ลด domain AdSense ใน CSP:** ไม่ทำ

| ไฟล์ | ขั้นที่ | รายการ |

## ลำดับการดำเนินการหลังอนุมัติ

1. **ขั้นที่ 1** — แก้ลิงก์ 1 จุด + ตั้งแคช static  
2. **ขั้นที่ 2** — ลด unused JS (dynamic import เฉพาะที่ปลอดภัย)  
3. **ขั้นที่ 3** — ปรับ LCP (attribute หรือพึ่งผลจากขั้น 2)  
4. **ขั้นที่ 4** — แก้คอนโซล/Issues จากโค้ดเรา  
5. **ขั้นที่ 5** — CSP/COOP (ทำหลังขั้น 1–4 และทดสอบ AdSense + Clerk ให้มั่นใจ)

---

## การยืนยันหลังทำแต่ละขั้น

- Build: `npm run build` ผ่าน
- หน้าที่สำคัญ: หน้าแรก, /tools, /pricing, sign-in, ใช้เครื่องมือ (generate) ทำงาน
- AdSense: ไม่เปลี่ยนการโหลดสคริปต์และไม่ลบ domain ใน CSP — โฆษณาไม่พัง
- Clerk: Sign-in / Sign-up / Sign-out ทำงาน
